# WebSocket Trade — план реализации

**Цель:** заменить REST-вызовы для выставления/отмены ордеров на постоянное WS-соединение (`/v5/trade`) с поддержкой batch-ордеров.

**Мотивация:** при большом количестве аккаунтов каждый REST-запрос открывает новое TCP-соединение. WS trade даёт одно постоянное соединение на аккаунт + возможность выставить все уровни сетки одним сообщением.

---

## Архитектура

```
AccountRunner
  ├── PrivateStream  (/v5/private)  — получение событий [уже есть]
  └── TradeStream    (/v5/trade)    — выставление ордеров [новое]
```

Каждый `AccountRunner` держит два WS на аккаунт.

---

## Что поддерживает /v5/trade

| Операция            | WS op                | Аналог REST                  |
|---------------------|----------------------|------------------------------|
| Выставить ордер     | `order.create`       | POST /v5/order/create        |
| Batch-выставление   | `order.create-batch` | POST /v5/order/create-batch  |
| Отменить ордер      | `order.cancel`       | POST /v5/order/cancel        |
| Отменить все        | `order.cancel-all`   | POST /v5/order/cancel-all    |
| Изменить ордер      | `order.amend`        | POST /v5/order/amend         |

Batch: до 20 ордеров за одно сообщение — весь стартовый цикл сетки в одну поездку.

## Ключевые задачи

### 1. `pkg/trader/trade_ws.go` — новый клиент
- Подключение к `wss://stream.bybit.com/v5/trade`
- Аутентификация (аналогично private stream — HMAC-SHA256)
- Отправка запросов с `req_id` и ожидание ответа (map[req_id]chan result)
- Авто-реконнект с backoff
- Методы: `PlaceOrder`, `PlaceOrderBatch`, `CancelOrder`, `CancelAllOrders`

### 2. `pkg/strategy/cycle.go` — использовать TradeStream
- `startCycle` → `PlaceOrderBatch` для всех уровней сетки
- `placeNextLevels` → `PlaceOrderBatch` для пакета уровней
- `cancelPlacedLevels` → `CancelAllOrders` через WS (уже через REST, переключить)
- `closeCycle` TP/SL отмены → через WS

### 3. `pkg/strategy/engine.go` — lifecycle TradeStream
- `AccountRunner.run`: запускать TradeStream параллельно с PrivateStream
- Передавать TradeStream в StrategyRunner

### 4. Fallback
- Если TradeStream не подключён — fallback на REST (текущая реализация)
- Логировать использование fallback

### 5. `services/api-gateway/trader_handler.go`
- `TraderPlaceOrder` (ручной ордер с терминала) — тоже переключить на WS trade

## Протокол сообщений

**Запрос:**
```json
{
  "reqId": "uuid",
  "header": { "X-BAPI-TIMESTAMP": "...", "X-BAPI-SIGN": "..." },
  "op": "order.create",
  "args": [{ "symbol": "BTCUSDT", "category": "linear", ... }]
}
```

**Ответ:**
```json
{
  "reqId": "uuid",
  "retCode": 0,
  "retMsg": "OK",
  "op": "order.create",
  "data": { "orderId": "...", "orderLinkId": "..." }
}
```

## Ожидаемый эффект

| Метрика              | Сейчас (REST)       | После (WS)            |
|----------------------|---------------------|-----------------------|
| Соединений на аккаунт | N запросов/сек     | 1 постоянный сокет    |
| Старт сетки (10 ур.) | 10 HTTP round-trips | 1 WS сообщение        |
| Отмена всех ордеров  | 1 REST (cancel-all) | 1 WS сообщение        |
| Латентность ордера   | ~50–150ms           | ~10–30ms              |
