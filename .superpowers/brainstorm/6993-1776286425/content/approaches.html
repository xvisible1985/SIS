3 архитектурных подхода

Выбери направление — детали обсудим в терминале

A

Монолит (Modular Monolith)

Один Go-бинарник с чёткими внутренними модулями: ingestion, signal engine, backtesting, API, WebSocket, webhook dispatcher. Всё в одном процессе, общая память.

Плюсы

  • Быстрый старт
  • Простой деплой
  • Нет сетевых задержек между компонентами

Минусы

  • Бэктестинг может блокировать real-time
  • Тяжело масштабировать отдельные части
  • Вырастет в сложности быстро
B

⭐ Сервисный монорепо (рекомендую)

4 отдельных Go-сервиса в одном репозитории: API Gateway, Data Ingester, Signal Engine, Webhook Dispatcher. Общаются через Redis Streams. React фронтенд отдельно.

Плюсы

  • Каждый сервис масштабируется независимо
  • Бэктестинг изолирован от real-time
  • Один репозиторий — просто разрабатывать
  • Redis уже нужен для кэша и pub/sub

Минусы

  • Чуть сложнее первый деплой
  • Нужно следить за совместимостью интерфейсов
C

Полные микросервисы + Kafka

Каждый компонент — отдельный сервис, общение через Kafka. Kubernetes для оркестрации. Максимальная масштабируемость с первого дня.

Плюсы

  • Максимальная отказоустойчивость
  • Горизонтальное масштабирование без ограничений
  • Enterprise-grade с первого дня

Минусы

  • Избыточно для старта SaaS
  • Kafka сложна в операционке
  • Долго разрабатывать
  • Высокие расходы на инфраструктуру
